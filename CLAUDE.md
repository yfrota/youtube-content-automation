# YouTube Content Automation Platform — Project Context

@AGENTS.md

Read this before starting work in a new session — it has the architecture decisions, schema, and gotchas that aren't obvious from the code alone. Full product pitch/status table lives in `README.md`; this file is the engineering context.

## What this is

Multi-tenant SaaS that automates YouTube content production (script rewriting, SEO, thumbnails) with AI agents and a RAG pipeline over the channel's existing video catalog. Built to expand to Instagram/Facebook/LinkedIn later — every tenant-scoped table already carries `client_id` and every content table carries `platform`, even though only YouTube is implemented today.

Built as part of a boutique AI automation consultancy (see README footer).

## Status

The full slice — index a real YouTube channel's catalog into RAG → generate a script with Script Forge → confirm it's genuinely cross-referenced against that catalog — has been run end-to-end against the live Supabase project, live Gemini, and live OpenRouter (not mocked). It passed, after fixing two real bugs that only showed up by actually calling the live APIs (see "Provider history" and the type gotchas below). If you're picking this up fresh, the pipeline works; you don't need to re-derive that from the code.

## Stack

Next.js 16 (App Router, Turbopack) + TypeScript + Tailwind + Supabase (Postgres + pgvector) + OpenRouter (agents) + Gemini embeddings. No test framework is set up — verification is manual (`npm run build` / `tsc --noEmit` + curl against the API routes).

**Provider history**: this started on Anthropic Claude (direct SDK) + OpenAI embeddings, then migrated to OpenRouter + Gemini (see `0003_update_embedding_dimension.sql` and the `feat: migrate embeddings to Gemini and agents to OpenRouter` commit). If you see stray mentions of Anthropic/OpenAI elsewhere (old commit messages, README's stack table), that's expected drift from before the migration, not a sign the migration is incomplete.

**Current model ids (verified live, not just "should work"):** embeddings use `gemini-embedding-001` (not `text-embedding-004` — Google retired it; the migration to `gemini-embedding-001` happened in code only, no new SQL migration needed since the schema only cares about the 768-dim output, not the model name). Script Forge uses `google/gemini-2.5-flash` via OpenRouter (not `gemini-flash-1.5` — OpenRouter retired it, "No endpoints found"). **Both of these are exactly the kind of thing that silently goes stale** — the provider can rename/retire a model id at any time, `tsc`/`lint`/`build` will never catch it, and the first sign is a runtime 404 from the live API. If something that used to work now 404s with "model not found" or "no endpoints found," check the provider's live `/models` list before assuming it's a code bug.

**Next.js 16 has real breaking changes vs. training data.** Check `node_modules/next/dist/docs/` before writing App Router code that looks "off."

## Architecture principles (don't violate these)

- **Multi-tenant**: every tenant-scoped table has `client_id` directly (denormalized, not just reachable via a join) — `clients` itself is the tenant root and has no `client_id`.
- **Multi-platform**: `projects`/`scripts`/`seo`/`thumbnails` carry `platform` (`youtube | instagram | facebook | linkedin`) directly, so the connector layer can dispatch without joining back to `projects`.
- **Connectors are adapters**: `lib/connectors/types.ts` defines `PlatformConnector` (`fetchCatalog`); each platform implements it (`lib/connectors/youtube.ts` today). No platform-specific branching outside the connector files.
- **Agents are platform-agnostic**: `lib/agents/` takes content/metadata + a `platform` tag and never branches on it. Platform-specific output formatting is the connector's job, not the agent's.

## Database (`supabase/migrations/`)

- `0001_initial_schema.sql` — `clients`, `projects`, `scripts`, `seo`, `thumbnails`, `approval_events`. Enums: `platform_type`, `approval_status` (`draft → kelly_review → client_review → approved → published`, matches README's approval flow — `kelly_review` is an internal review step, not a typo). `scripts.embedding` is `vector(1536)` with an HNSW index. **RLS is enabled on every table with no policies yet** — there's no auth/tenant-membership model, so the only way to read/write right now is `lib/supabase/admin.ts` (service-role key, bypasses RLS).
- `0002_catalog_indexing_support.sql` — adds:
  - `unique index on projects (client_id, external_video_id)` — **not partial**. NULLs never collide for uniqueness, so manually-created in-production projects (no video id yet) coexist fine; only real catalog video ids dedupe. Makes catalog re-indexing an idempotent upsert.
  - `unique index on scripts (project_id) where raw_transcript is null` — **partial**, catalog entries only. Real Script Forge drafts always set `raw_transcript`, so multi-version script generation per project is never blocked by this.
  - `match_scripts(query_embedding, match_client_id, match_platform, match_count)` — Postgres function exposed as an RPC, because supabase-js's query builder can't express `order by embedding <=> $param`. This is the only way RAG search works.
- `0003_update_embedding_dimension.sql` — drops/recreates `scripts.embedding` as `vector(768)` (was `vector(1536)`, OpenAI-sized) since the embedding provider moved to Gemini. Also redefines `match_scripts` with a `vector(768)` parameter — **if you ever change the embedding dimension again, you must update both the column and this function in the same migration**, or RAG search breaks with a dimension mismatch on the `<=>` operator. Includes a defensive `drop function` before the `create or replace`, since changing a function's parameter typmod is borderline enough behavior in Postgres that it's not worth relying on.
- `0004_grant_service_role_privileges.sql` — grants `service_role` `select/insert/update/delete` on all six tables, plus an `alter default privileges` rule so future tables auto-grant too. **This was a real production-blocking bug**, not a precaution: `service_role`'s `BYPASSRLS` only skips row-level policies — it does not imply ordinary table grants, and none of `0001`–`0003` ever granted any. Every API route was returning "permission denied for table X" until this was applied, discovered by actually running the app against the live database (RLS-focused reasoning in `0001`'s comments and `docs/rls-policies.md` talks about policies, not base grants — don't conflate the two layers again).
- `0005_seo_and_keywords.sql` — adds `scripts.keywords_context` (jsonb, the keyword-research selection passed to the SEO Engine) and `seo.titles`/`tags`/`hashtags`/`selected_title` (the original `0001` columns `title_options`/`keywords` are unused dead weight now, left in place, not dropped), plus `unique index on seo (project_id)` so the SEO Engine can upsert with `onConflict: "project_id"`. **Until this is applied, `GET /api/projects/[id]` 500s entirely** (it selects `keywords_context`) — not just a missing-feature gap, the whole detail page goes down. Verified this exact failure live against a pre-migration database before the fix.
- `0006_project_language.sql` — adds `projects.language text not null default 'pt-BR'` (only `'pt-BR'`/`'en-US'` are real options — enforced at the application layer, e.g. the new-project form's `<select>`, not a Postgres enum/check). **Same failure pattern as `0005`**: `GET /api/projects/[id]`, `POST /api/projects`, and the Script Forge/SEO Engine routes all select or insert `language` directly, so they break (not just degrade) until this is applied. Verified live before and after.
- `0007_match_scripts_catalog_titles.sql` — redefines `match_scripts` again (drop-then-recreate, same reason as `0003`: changing a table-returning function's output columns isn't something `create or replace` can do in place), bundling two real bugs fixed together because both needed the same join:
  - Joins `projects` and returns `title` — previously `match_scripts` only returned `project_id` (a uuid), and the agent prompts (`script-forge.ts`/`seo-engine.ts`) interpolated that raw uuid into the "related existing videos" context block. Verified live that the model sometimes echoed the uuid into generated script content (one pre-fix script in the database has a literal project_id string in its `content`). Agents now reference matches by `m.title`.
  - Adds `p.external_video_id is not null` to the `where` clause, **before** the `order by`/`limit` — manually-created in-production projects (the dashboard's "Novo projeto" form, `external_video_id` always null) aren't real published videos and must never be offered as a cross-reference candidate; only catalog-indexed videos (via `POST /api/connectors/youtube/index`) are eligible. Filtering here rather than in application code after the RPC call matters: post-filtering could silently return fewer than `match_count` results even when enough eligible catalog matches exist further down the similarity ranking. Verified live: with only manual-project scripts embedded (no catalog indexed for the test client), a same-topic Script Forge call returned `crossReferencedProjectIds: []` — pre-fix this would have matched them.
- `0008_clients_profile_and_project_fields.sql` — adds `clients.image_url`/`description`/`phone` (a client-management UI's profile fields) and `projects.priority`/`deadline`/`tags` (dashboard filtering/sorting). **`clients.email` was deliberately not added** — `clients.contact_email` already existed (`0001`) and serves the same purpose; the API/TS layer exposes it as `email` (`toClientProfile()` in `lib/dashboard/types.ts` maps `contact_email` ↔ `email`) rather than creating a second, overlapping column. `priority` is a plain `text` column (`'low'|'normal'|'high'|'urgent'`) constrained at the application layer only, same precedent as `language` (`0006`) — not a Postgres enum/check. `clients.updated_at`/`projects.updated_at`/`created_at` were **not** touched — they already existed since `0001` with `not null default now()` and an auto-update trigger (`set_updated_at()`); re-adding them would've been a silent no-op.
- `0009_platform_type_spotify_tiktok.sql` — `alter type platform_type add value` for `'spotify'`/`'tiktok'` (each its own statement — Postgres doesn't allow adding an enum value in the same transaction as other schema work that uses it, though that wasn't a concern here since this migration does nothing else). Brings the enum to 6 values; `Platform`/`PlatformType` were widened to match in `lib/connectors/types.ts`, `lib/dashboard/types.ts`, and `lib/supabase/database.types.ts` (all three needed updating together or assigning a real `platform: "spotify"` row into a `Project`/`ProjectDetail` would fail to type-check) — see "What's implemented" below for the rest.
- `0010_content_type_and_deliverables.sql` — adds `projects.content_type` (`'youtube_tutorial'|'podcast_vodcast'|'short_form'`, `not null default 'youtube_tutorial'`, app-layer-only constraint, same precedent as `language`/`priority`) and, denormalized onto `scripts` (same pattern as `client_id`/`platform` across this schema — records what type a specific script was generated under, independent of whatever the project's `content_type` is *now*): `content_type` (same default) plus three nullable text columns — `clip_script`, `cta_line`, `pod_description` — the `podcast_vodcast`-only deliverables Script Forge's tool can optionally fill (see "What's implemented" below).
- Next migration should be `0011_*.sql` — don't renumber existing ones.

## What's implemented

- `lib/connectors/youtube.ts` — **metadata-only via `YOUTUBE_API_KEY`**, plain `fetch` (no `googleapis`). Pulls title/description/tags via `channels.list` → `playlistItems.list` (paginated) → `videos.list` (batched 50). **Deliberately no OAuth, no caption/transcript fetching** — explicit decision to avoid standing up an OAuth consent flow + token-storage table before it's needed (e.g. for publishing or private-video access later). **`channels.list`'s `id` param only accepts a literal Channel ID (`UCxxxx...`, 24 chars) — a `@handle` or full URL passed there returns zero items, silently, no error.** `resolveChannelId(input)` (exported, called by `fetchCatalog` itself so any caller benefits) normalizes a Channel ID / bare `@handle` / full `youtube.com` URL (any of `/@handle`, `/channel/UCxxxx`, `/c/name`, `/user/name`, with or without `https://`/`www.`, trailing path/query) down to the canonical Channel ID before any other API call. Resolution order: already-canonical id (regex, no API call) → `/channel/UCxxxx` (regex-extracted, no API call) → `@handle` via `channels.list?forHandle=` (verified live) → legacy `/c/`/`/user/` name via `channels.list?forUsername=`, falling back to `search.list?type=channel` (100 quota units vs. 1 — only hit for that rare legacy case) if `forUsername` comes up empty. `POST /api/connectors/youtube/index` also resolves explicitly itself (`resolveChannelId` is exported, not just an internal helper) *before* calling `fetchCatalog`/`indexCatalog`, so the canonical id — not whatever format the caller sent — is what gets stored in `projects.external_channel_id`; otherwise re-indexing the same channel under a different input format each time would store a different `external_channel_id` per run even though `external_video_id` already dedupes the actual video rows. The `/projects/new` form's channel URL field (placeholder `https://www.youtube.com/@seucanal`) was already inviting `@handle`-style URLs before this existed — they previously silently indexed zero videos.
- `lib/rag/embeddings.ts` — Gemini `gemini-embedding-001`, 768 dims via `outputDimensionality` (matches the schema; don't change one without the other — and remember `match_scripts` needs updating too, see the `0003` note above). Plain `fetch` against `generativelanguage.googleapis.com`, same lightweight-no-SDK pattern as the YouTube connector. Uses `GEMINI_API_KEY`.
- `lib/rag/indexer.ts` — `indexCatalog()`. Upserts `projects` via `.upsert(..., {onConflict: "client_id,external_video_id"})` (safe, non-partial index). For `scripts`, **does NOT use `.upsert()`** — does an explicit select (`project_id` + `raw_transcript is null`) then insert/update, because PostgREST's upsert can't target a partial unique index (no way to attach the `WHERE` clause). If you're tempted to "simplify" this to `.upsert()`, don't — it'll fail at runtime with "no unique or exclusion constraint matching the ON CONFLICT specification."
  - **Vercel timeout fix (live-diagnosed, not guessed)**: indexing 76 videos in one call was 502-ing on Vercel (function timeout). Added temporary `console.time` instrumentation and measured live against a real 76-video slice: `embedBatch(76)` — one batched Gemini call for the whole input — took **1.7s, not the bottleneck**. The per-video loop (upsert `projects` → select `scripts` → insert-or-update `scripts`, run fully sequentially) took **44s total, ~550ms per video** — that's the real cost, 3 sequential Supabase round-trips × 76 videos with zero concurrency. Fixed by running up to `PER_VIDEO_CONCURRENCY` (10) of these per-video chains concurrently via a small hand-rolled `runWithConcurrency` (no `p-limit` dependency) instead of the old plain `for` loop — `indexed++`/`failed.push()` are safe to mutate from concurrent workers since Node's single-threaded event loop means no true parallel write. Live-verified: same 76 videos in one call dropped from 44s to **9.6s** — a real improvement, but still close enough to a 10s ceiling that concurrency alone isn't a safe fix by itself (see below). `EMBED_BATCH_SIZE` (100) was left unchanged since `embedBatch` was never the bottleneck.
- `app/api/connectors/youtube/index/route.ts` now also paginates via **`?limit=` (default 10)** — independent of the concurrency fix above, since neither one alone scales to arbitrarily large channels (a 9.6s call for 76 videos leaves uncomfortably little margin under any serverless timeout, and a channel with hundreds/thousands of videos would still run long even at the faster per-video rate). Each call: fetches the *full* catalog from `fetchCatalog` (cheap — metadata only, ~2s for 76 videos in testing), queries `projects` for which of those `external_video_id`s already exist for `clientId` (not an `offset` the caller tracks — checking the DB directly means a call that fails partway, or any retry, is automatically picked up correctly by the next call with no manual resume bookkeeping), then indexes only the next `limit` not-yet-indexed videos. Returns `{ totalVideos, indexed, remaining, failed }` — `totalVideos` is the full catalog size, not the current slice's size. Callers (e.g. a future "Reindexar" UI) are expected to call this repeatedly with the same body until `remaining === 0`. Live-verified across 3 sequential calls against a real 171-video channel: call 1 `{indexed:10, remaining:161}` in 4.6s, call 2 (same request body) correctly resumed at `{indexed:10, remaining:151}` with no duplicate work, call 3 with `?limit=76` `{indexed:76, remaining:75}` in 9.6s (matches the concurrency-fix number above, confirming both fixes compose correctly). **Vercel plan wasn't confirmed at the time of this fix** (no `.vercel/`/`vercel.json` in this repo, CLI not installed/authenticated, user didn't know offhand) — this pagination approach was deliberately chosen because it's correct regardless of plan or `maxDuration` support; `export const maxDuration = 60` was *not* added speculatively (only meaningful on Pro+, unverified here) but is a safe, low-risk thing to add later once the plan is confirmed — it would let each call index more before needing to paginate, not replace the pagination itself for large channels.
- `lib/rag/search.ts` — `searchCatalog()`, wraps the `match_scripts` RPC. Returns `RagMatch[]` including `title` (the matched video's project title, joined inside `match_scripts` itself as of `0007` — see the Database section) — agent prompts must reference matches by `title`, never `projectId`, or the model echoes a raw uuid into generated content (this happened live, pre-fix). `match_scripts` also only ever returns catalog-indexed videos (`external_video_id is not null`) as of `0007`, so callers never need to filter manual in-production projects out themselves.
- `lib/agents/script-forge.ts` — OpenRouter (`https://openrouter.ai/api/v1`, `OPENROUTER_API_KEY`), model `google/gemini-2.5-flash` by default, `max_tokens: 12000` set explicitly (raised from `4096` → `8192` for `podcast_vodcast`'s full-episode verbatim scripts, `0010`). **That `max_tokens` is load-bearing, don't drop it**: omit it and OpenRouter defaults to the model's max output (65535 for this one), which fails with a `402` "requires more credits, upgrade to a paid account" if the account's balance can't cover it — reads like a billing problem, not a missing-parameter bug, and cost real debugging time to trace back. **The 65535 ceiling is the model's, not the account's** — tried bumping to `16384` and hit a live `402` anyway: "requested up to 16384, but can only afford 13146," i.e. the OpenRouter account's *current credit balance* is the binding constraint, independent of the model's own cap. `12000` is the real, currently-affordable value verified live (with margin below the 13146 ceiling observed at test time) — it'll get quiet headroom as credits are topped up, but don't bump this constant again without a live test; the model's 65535 ceiling tells you nothing about what the account can currently afford. Uses the `openai` npm package pointed at OpenRouter's base URL (the standard OpenRouter integration pattern — its API mirrors OpenAI's regardless of which underlying model it proxies to). **Forced tool-use** via `tool_choice: "required"` (not a named-function force — "required" is more broadly supported across the models OpenRouter can route to, and equivalent here since `emit_script` is the only tool defined) to get structured `{content, hook, chapters}` back, plus three optional tool properties as of `0010` (`clip_script`/`cta_line`/`pod_description`, never in `required` — only `podcast_vodcast`'s prompt asks the model to fill them, so they stay `null` for `youtube_tutorial`/`short_form`). Pulls RAG context from `searchCatalog` before generating, building a `contextBlock` that lists each match by `m.title` plus an explicit "use the TITLE, never an id" instruction in the prompt — see the `0007` migration note for why this changed. `llm_provider` on the saved row stores the actual model id (e.g. `"google/gemini-2.5-flash"`), not the gateway name. **Two prompts as of `0010`** (`buildPrompt()` picks one by `contentType`): `youtube_tutorial`/`short_form` share the original prompt verbatim (`short_form` has no dedicated structure yet — it's "futuro" per the new-project form's own option label); `podcast_vodcast` gets a from-scratch prompt for a full-episode verbatim script with a fixed 7-part show structure (teaser hook → formal intro → roadmap → development → mid-episode CTA → closing questions → sign-off) in a warm/philosophical voice, deliberately not this agent's other, more corporate-tutorial tone. `runScriptForge`'s input/output (`ScriptForgeInput`/`ScriptForgeOutput` in `lib/agents/types.ts`) both gained `contentType`; output also gained `clipScript`/`ctaLine`/`podDescription` (all three persisted onto the `scripts` row alongside `content_type`, denormalized from the project's own `content_type` at generation time).
  - **`podcast_vodcast` prompt size calibration (prompt-only fix, no migration)**: the prompt now computes `transcriptWordCount` from the raw transcript and tells the model the final script should land within 80%-120% of that count, plus appends "desenvolva completamente — não use placeholder ou resumo" to every one of the 7 structure sections (so sections don't get short-changed while the model chases the word target). **Live-tested caveat, not yet fully resolved**: against a short (332-word) synthetic test transcript, the model still produced 948 words — nearly 3x the 266-398 target — because the mandatory 7-section full-episode structure (teaser hook + formal intro + roadmap + 2-3 long development parts + mid-episode CTA + closing questions + sign-off, each told to "develop completely") structurally needs more raw content than a tight word ceiling allows for a short input. The size instruction and the per-section "develop completely" instructions are in real tension for short inputs; this wasn't re-tuned further since both instructions were the user's own explicit spec. **Re-test with a real, full-length podcast transcript (thousands of words, not a short synthetic one)** before concluding this calibration doesn't work — the original bug (truncation/over-summarization) was observed on long real transcripts, where the structural floor and the 80%-120% ceiling may not conflict the way they did in this short test.
  - **Language mandate**: an "IDIOMA OBRIGATÓRIO" instruction now sits at the very top of the `podcast_vodcast` prompt, before anything else, explicitly covering the main script *and* `clip_script`/`cta_line`/`pod_description` — previously only the main script had an explicit language instruction (injected mid-prompt), so the three extra deliverables could drift to the wrong language even when the main script was correct. Verified live with an `en-US` project: all four fields came back in English.
- API routes: `GET /api/clients` (list, ordered by name, each row includes a computed `projectsCount` — one aggregate `select client_id from projects` query mapped into counts, not N+1), `POST /api/clients` (create — `slug` is generated server-side via a basic `slugify(name)`, retried once with a random suffix on a unique-constraint collision since callers never see/set `slug` directly), `GET /api/clients/[id]` (profile + `projectsCount`), `PATCH /api/clients/[id]`, `DELETE /api/clients/[id]` (**blocked with a 400 if the client still has any projects** — `projects.client_id` is `on delete cascade`, so without this guard a delete would silently wipe every project/script/seo/thumbnail/approval_event the client owns).
  - `GET /api/projects` (the dashboard/client-detail list) — **`clientId` semantics changed**: a specific id scopes to that client (unchanged), but omitting it or passing the literal `"all"` now lists **across every client** (each project resolves its own `client` via a `Map`, no more one-shared-profile-for-the-whole-response). `"all"` exists as an explicit sentinel rather than just "param absent" because the dashboard always sends *something* and needs to tell "user picked 'todos os clientes'" apart from "param wasn't passed at all" — those used to be the same wire state and silently meant `DEV_CLIENT_ID`. **`DEV_CLIENT_ID` is no longer this route's default** — a list endpoint quietly scoping to one hidden tenant stopped making sense once `/clients` existed to browse all of them; `POST /api/projects` still falls back to it (now mostly vestigial, see below). Also still supports `?sort=name|created_at|updated_at|deadline|priority` + `?order=asc|desc` (default `updated_at desc`; `priority` sorted in application code by severity rank, see `lib/dashboard/types.ts`'s `Priority`, since alphabetical DB ordering would put "high" before "low" before "normal"), `?q=` title search, `?tag=` exact-tag filter via jsonb containment.
  - `POST /api/projects` (create — accepts `priority`/`deadline`/`tags`; `clientId` falls back to `DEV_CLIENT_ID` only if the caller omits it, but `/projects/new` always sends a real one now via its required client dropdown — see below).
  - `GET /api/projects/[id]`, `PATCH /api/projects/[id]`, `DELETE /api/projects/[id]` — **bug fixed: these three no longer filter by `client_id = DEV_CLIENT_ID`.** They used to (silently correct before the clients pages existed, since literally every project belonged to `DEV_CLIENT_ID` anyway) — caught live while verifying `/clients/[id]`: clicking into a project belonging to a freshly-created, non-`DEV_CLIENT_ID` client 404'd with "Project not found" even though the project obviously existed. `id` is already a unique, sufficient key for a single-resource lookup; no client scoping is needed or correct here once projects can belong to any client. `GET` also returns `client`/`channelUrl`/`priority`/`deadline`/`tags` (added earlier) and `PATCH` still only touches title/priority/deadline/tags (card editing, independent of the script/seo/thumbnail approval flow); `DELETE` still cascades to the project's own scripts/seo/thumbnails/approval_events (`on delete cascade` since `0001`) — that's the intended behavior, not a hazard.
  - `PATCH /api/scripts/[id]` (`status` and/or `keywordsContext`, independently — "Salvar keywords" patches only the latter, "Aprovar roteiro" patches both, "Desfazer aprovação" patches only `status` back to `draft`), `GET /api/scripts/[id]/keywords` (extracts SEO keywords from `script.content` via forced tool-use — `tool_choice: "required"`, tool `extract_keywords` — same pattern as `script-forge.ts`/`seo-engine.ts`; **not** a plain free-text completion, that was the original implementation and it intermittently threw JSON-parse errors when the model wrapped its output in markdown fences or added stray text), `GET /api/youtube/trending?q=` (Google's public autocomplete endpoint), `POST /api/agents/seo-engine`, `PATCH /api/seo/[id]` (`status` and/or `selectedTitle`), `POST /api/connectors/youtube/index`, `POST /api/agents/script-forge`. **None have auth/session checks** — each has a `// TODO(auth):` comment; this is accepted for now since they're server-only and there's no real tenant-membership model to check against yet.
- `app/api/youtube/trending/route.ts` — **the endpoint serves ISO-8859-1, not UTF-8, regardless of what its Content-Type header claims.** `Response.text()` always decodes as UTF-8 and silently mangles accented characters (verified live: "podcast cristão" came back as "podcast crist?o"). Decode via `new TextDecoder("iso-8859-1").decode(await res.arrayBuffer())` instead. Response body is JSONP (`window.google.ac.h([...])`) — parsed by slicing between the first `[` and last `]`, not by stripping a specific callback name, so it's robust either way.
- `lib/agents/seo-engine.ts` — same shape as `script-forge.ts` (OpenRouter, `google/gemini-2.5-flash`, `max_tokens: 4096`, `tool_choice: "required"`, `toolCall.type !== "function"` narrowing, same `m.title`-based RAG `contextBlock` as of `0007`). Takes `keywordsContext: string[]` and, if non-empty, adds a "KEYWORDS PRIORITÁRIAS" block to the prompt — verified live that the model actually weaves the supplied keywords into titles/tags, not just acknowledges them. `llmProvider` is an optional input (defaults to the module's `MODEL` constant) that lets a caller override which OpenRouter model generates that one call — the multi-provider-selector hook the README describes, scoped per-call rather than built out as real UI yet.

## Type gotchas (already hit these once — don't reintroduce them)

- `lib/supabase/database.types.ts` row types **must be `type` aliases, not `interface`s**. supabase-js 2.108.2 checks `Database['public']['Tables'][x]['Row'] extends Record<string, unknown>` structurally, and TypeScript only grants object type literals (and `type` aliases of them) an implicit index signature for that check — `interface` declarations are excluded from it and silently resolve every `.insert()`/`.upsert()` call to `never`. Same applies to `lib/agents/types.ts`'s `ScriptChapter` (used in a `jsonb` insert).
- The `openai` SDK types `ChatCompletion.choices[].message.tool_calls[]` as a union (`ChatCompletionMessageFunctionToolCall | ChatCompletionMessageCustomToolCall`, discriminated by `type: "function" | "custom"`). Narrow with `toolCall.type !== "function"` before accessing `.function` — accessing it directly on the union doesn't compile. See `script-forge.ts`'s tool-call handling.

## Verifying changes

No test framework — run `npx tsc --noEmit`, `npm run lint`, `npm run build` after any change to `lib/` or `app/api/`. None of those three would have caught either bug fixed above — both were live-API drift (retired model ids), invisible until something actually called the real endpoint.

**If every route suddenly 404s after running `npm run build`, that's `.next` cache drift, not a code bug**: `next build` and `next dev` share the `.next` directory, and a prod manifest left over from a build confuses the dev server's routing. Fix: `rm -rf .next` before restarting `npm run dev`.

For end-to-end behavior, this exact sequence has been run successfully against the live stack: `npm run dev` → create a `clients` row directly via the Supabase REST API (no client-creation route exists) → `POST /api/connectors/youtube/index` with a real channel → re-run it to confirm idempotency (counts shouldn't move) → `POST /api/projects` → `POST /api/agents/script-forge` with a transcript topically related to the indexed channel, confirm `crossReferencedProjectIds` is non-empty → run it again with an unrelated-topic transcript, confirm the ids differ. If you want quantitative proof RAG is real and not coincidental, call the `match_scripts` RPC directly with both query embeddings and compare `similarity` — expect ~0.78–0.84 for on-topic vs. ~0.48 for off-topic. Clean up test data afterward (`DELETE` the test `clients` row — cascades to everything).
